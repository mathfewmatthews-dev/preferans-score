# Архитектура Preferans Score

## Границы системы

- `app.js` — UI и существующие правила расчёта партии. Он формирует команды, но не пишет Firestore напрямую в production-пути.
- `src/domain/` — типы состояния, инварианты, хеширование, патчи и воспроизводимые события.
- `src/persistence/` — независимые реализации локального и удалённого хранилищ.
- `src/sync/` — очередь неподтверждённых событий, транзакционные ревизии и обнаружение конфликтов.
- `src/infrastructure/logger.ts` — диагностические сообщения с удалением имён, состояния, токенов и URL.
- `src/audit/` и `scripts/audit-firestore*.mjs` — read-only проверка снимков, scoreLog, хешей и цепочек событий.

Игровой журнал (`state.history`), технические события (`games/{id}/events`) и диагностические логи — разные данные. Они не используются друг вместо друга и не имеют общего хранилища.

## Инварианты сохранения

1. Каждая локальная партия имеет собственный ключ по `gameId`; основная страница не открывает shared autosave.
2. Перед заменой локального v2-снимка создаётся ограниченная backup-ревизия.
3. Неподтверждённые события сохраняются вместе с локальным снимком и переживают reload/offline.
4. Firestore принимает только следующую ревизию внутри транзакции. Повтор события с тем же `eventId` идемпотентен.
5. При одновременном изменении одного поля локальное событие остаётся pending и показывается конфликт; удалённый снимок его не затирает.
6. Legacy `json-v1` читается лениво и сохраняется в `backups/legacy-v1` при первой v2-транзакции.

## Формат Firestore v2

`games/{gameId}` хранит текущий снимок, номер ревизии и SHA-256 состояния. Неизменяемые события находятся в `games/{gameId}/events/{eventId}`. Это позволяет отдельно проверять текущий снимок и воспроизведение цепочки операций.

Новые правила находятся в `firestore.rules`; `firebase.json` и `.firebaserc` связывают их с проектом `preferans-score`. Публикация правил — отдельная операция от деплоя GitHub Pages.

## Проверки

```text
npm run check
npm run test:e2e:audit
npm run test:e2e:firebase
npm run audit:database:public -- <gameId> [gameId...]
npm run audit:database
```

`audit:database:public` читает только явно указанные партии через публичные Rules. Полный обход коллекции выполняется `audit:database` с read-only service account из `GOOGLE_APPLICATION_CREDENTIALS`; credentials запрещено хранить в репозитории.

## Публикация

`.github/workflows/pages.yml` проверяет типы, линтер, unit-тесты и production build, затем публикует только `dist/`. В настройках GitHub Pages источником должен быть выбран **GitHub Actions**.
